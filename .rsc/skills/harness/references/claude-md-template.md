## Working rules

- Before touching a subproject, run `git -C <subproject> status --short`.
  The user may have in-flight changes; don't revert or merge them without
  understanding them.
- Ignore generated outputs unless the user asks for them: `node_modules/`,
  `.next/`, `.venv/`, `.pytest_cache/`, `.ruff_cache/`, `.mypy_cache/`,
  `.dart_tool/`, `build/`, `__pycache__/`.
- Keep subproject boundaries clean. Don't mix dependencies or stacks
  across repos.
- Never publish secrets. `01-TOOLS/**/.env`, `CREDENTIALS.md`, SSH keys,
  service accounts, and tokens are sensitive even though they live in the
  workspace.
- For UI changes, read the canonical design document for the affected
  surface before editing.
- For documentation, update `02-DOCS/` following the embedded wiki
  protocol: new sources go in `raw/<topic>/`, distilled knowledge in
  `wiki/<topic>/`. The skill maintains `wiki/index.md` and `wiki/log.md`.
